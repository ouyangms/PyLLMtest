针对 **2000+ 车控技能** 的场景，`example_queries`（典型用户问法）的质量直接决定了 **向量检索（RAG）的召回率**。如果这里覆盖不足，后续的 LLM 再强也“无米下锅”。

对于每个技能，推荐准备 **5-8 条** 高质量的 `example_queries`。这 5-8 条必须覆盖以下 **5 个维度** 的口语化变体：

### 1. 推荐的 5 个维度模板

假设技能是：**`window_open_driver_slit` (主驾车窗开一条缝)**

| 维度 | 描述 | 示例 (Example Queries) | 作用 |
| :--- | :--- | :--- | :--- |
| **A. 标准指令型** | 用户清晰、完整地表达意图，接近技能定义。 | 1. “打开主驾窗户一条缝”2. “把驾驶员侧车窗开一点” | 覆盖最基础的匹配，保证下限。 |
| **B. 场景/感受型** | 用户不直接说动作，而是描述**状态**或**感受**（隐式意图）。 | 3. “主驾有点闷，透透气”4. “车里烟味太大了，散散味”5. “太热了，开个缝凉快一下” | **最关键！** 捕捉真实用户的自然表达，解决“没说动词”的问题。 |
| **C. 模糊/口语型** | 省略主语、使用方言词汇、叠词或极简表达。 | 6. “窗户开条缝”7. “主驾窗开个小口”8. “留个缝儿” | 覆盖日常随意的说话习惯，提高鲁棒性。 |
| **D. 否定/反向型** | (可选) 如果该技能有明确的反向操作，可在此提及边界。 | 9. “别全开，就开一点点” | 帮助模型区分“全开”和“开缝”的边界。 |
| **E. 组合/上下文型** | 模拟多轮对话或组合指令的前半部分。 | 10. “先把主驾窗户...” (需配合上下文)*(注：通常单独存储，或作为特殊样本)* | 增强对不完整句子的容错。 |

### 2. 具体案例演示

为了让你更直观地理解，以下是针对三个不同类别技能的 **推荐写法清单**：

#### 案例一：复杂参数类技能
**技能 ID**: `ac_set_temp_zone_driver` (设置主驾空调温度为 24 度)
*注意：这里不写具体数字，而是写“调节温度”的意图，具体数字由 LLM 从语音中提取。*

```json
"example_queries": [
  "主驾空调调到 24 度",             // A. 标准 (带具体参数)
  "把驾驶员这边温度设成 24",        // A. 标准 (同义替换)
  "主驾有点冷，调到 24 度",         // B. 场景 + 意图
  "空调 24 度，只要主驾",           // C. 倒装/口语
  "温度太高了，降到 24 度",         // B. 感受型 (隐含调低)
  "设为 24 度",                     // C. 极简 (依赖上下文默认主驾)
  "帮我把主驾温度弄到 24",          // C. 礼貌/口语化动词
  "24 度，主驾区域"                 // C. 关键词堆砌
]
```

#### 案例二：模糊动作类技能
**技能 ID**: `mirror_fold_all` (折叠所有后视镜)

```json
"example_queries": [
  "折叠后视镜",                     // A. 标准
  "把镜子收起来",                   // B. 生活化用语 ("收起来" vs "折叠")
  "锁车自动折镜",                   // B. 场景关联 (虽然可能是设置项，但用户常这样喊)
  "镜子太宽了，收一下",             // B. 原因驱动
  "收起所有后视镜",                 // A. 强调范围
  "把耳朵收起来",                   // C. 拟人化/俗称 ("耳朵")
  "外后视镜折叠",                   // A. 书面语
  "关门时把镜子折了"                // B. 场景驱动
]
```

#### 案例三：多义词消歧类技能
**技能 ID**: `light_read_on` (打开阅读灯) vs `light_cabin_on` (打开车内氛围灯/顶灯)
*此处需要特别强调“阅读”这一特定场景，防止检索到通用顶灯。*

```json
"example_queries": [
  "打开阅读灯",                     // A. 标准
  "我要看书，开个灯",               // B. 场景驱动 (关键区分点)
  "主驾头顶那个看书的灯",           // C. 位置 + 功能描述
  "开一下阅读灯，太黑了",           // B. 场景
  "照亮书本的灯",                   // B. 功能描述
  "阅读灯亮一点",                   // A. 变体
  "帮我开个小灯看书",               // C. 口语
  "头顶的阅读灯"                    // C. 位置限定
]
```

### 3. 数据构建的“避坑”指南

在准备这 2000+ 技能的 `example_queries` 时，请务必遵守以下原则：

1.  **拒绝“机器人语言”**：
    *   ❌ 错误：“执行主驾车窗开启 10% 操作”
    *   ✅ 正确：“窗户开条缝”、“透透气”
    *   *原因*：用户不会像写代码一样说话。向量模型需要学习的是**人类自然语言的分布**。

2.  **覆盖“同义词”和“俗称”**：
    *   车窗 -> 玻璃、窗户
    *   后备箱 -> 尾箱、后盖
    *   空调 -> 冷气、暖气、风
    *   后视镜 -> 镜子、耳朵
    *   *建议*：建立一个**车控领域同义词库**，在生成例句时自动替换。

3.  **处理“指代不明”**：
    *   很多指令省略了“主驾”、“左前”等位置信息。
    *   例句中必须包含大量**省略位置**的样本（如“开窗”），让检索系统能将其召回，然后交给 LLM 结合当前座位状态（如“主驾正在说话”）来推断具体位置。

4.  **数量平衡**：
    *   不要所有技能都写 20 条，也不要只写 1 条。
    *   **高频技能** (空调、车窗、导航)：每个技能 **8-10 条**，覆盖各种刁钻说法。
    *   **低频技能** (座椅按摩特定模式、工程模式)：每个技能 **3-5 条**，覆盖标准说法即可。

### 4. 自动化生成建议 (提效工具)

手动写 2000 * 5 = 10,000 条数据太累。建议使用 **云端大模型 (如 Gemma 3 27B 或 Claude 3.5)** 批量生成：

**Prompt 示例**:
```text
你是一个车载语音数据专家。
任务：为技能 "{skill_description}" 生成 6 条用户可能说的口语化指令。
要求：
1. 包含 1 条标准指令。
2. 包含 2 条描述感受/场景的指令 (如"太热了")。
3. 包含 2 条极简/口语化指令 (如"开条缝")。
4. 包含 1 条带有俗称/同义词的指令。
5. 不要出现技能 ID 或技术参数，完全模拟人类口吻。
6. 输出 JSON 列表格式。

技能描述: {skill_description}
```

通过这种方式，你可以在 1 小时内生成高质量的 2000+ 技能语料库，直接用于构建 FAISS 索引。

### 总结
对于每个技能，**5-8 条** 精心设计的 `example_queries` 是性价比最高的选择。重点在于**覆盖“场景感受”和“口语俗称”**，这将直接决定你的系统在真实路况下的**召回率**能否达到 95% 以上。
