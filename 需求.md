针对**2000+ 车控技能**在**高通8295**上的落地，以下是**工业级**的详细实施步骤。本方案基于“路由 + 检索 + 生成”的三级架构，确保延迟 <1s 且准确率 >95%。

---

### 第一阶段：数据工程与技能库构建 (离线准备)

这是最关键的一步。不要直接把 2000 个技能丢给模型，必须建立结构化的“技能知识图谱”。

#### 1. 技能标准化定义 (Skill Schema)
为每个技能定义唯一的 ID、自然语言描述、参数规范和典型用户问法（Query）。
*   **文件结构**：`skills_database.json`
```json
[
  {
    "skill_id": "climate_ac_on_driver",
    "category": "climate_control", 
    "domain": "hvac",
    "description": "打开主驾驶侧空调制冷",
    "params_schema": { "type": "object", "properties": { "temp": {"type": "number"}, "fan_speed": {"type": "string"} } },
    "example_queries": [
      "主驾太热了",
      "打开驾驶员空调",
      "冷风开大点"
    ],
    "execution_code": "HVAC_SET(zone=driver, state=on, mode=cool)"
  },
  {
    "skill_id": "window_open_left_slit",
    "category": "window_control",
    "domain": "body",
    "description": "将左前车窗开启一条缝隙 (约10%)",
    "params_schema": { "type": "object", "properties": { "percentage": {"type": "number", "enum": [10]} } },
    "example_queries": [
      "窗户开个缝",
      "透透气",
      "左窗开一点"
    ],
    "execution_code": "WINDOW_SET(zone=left_front, position=10%)"
  }
  // ... 共 2000+ 条
]
```
*   **关键点**：`category` 字段用于第一级路由；`example_queries` 用于增强检索召回率。

#### 2. 构建向量索引 (Vector Index)
使用轻量级 Embedding 模型将文本转化为向量，建立本地检索库。
*   **模型选择**：`bge-small-zh-v1.5` (中文效果好，仅 ~30MB) 或 `all-MiniLM-L6-v2`。
*   **工具**：`FAISS` (Facebook AI Similarity Search) 或 `ScaNN`。
*   **操作步骤**：
    1.  **文本拼接**：将 `description` + `example_queries` 拼接为长文本。
    2.  **向量化**：遍历 2000+ 技能，生成 2000+ 个 384 维向量。
    3.  **建库**：保存为 `skill_index.faiss` 和 `skill_mapping.pkl` (记录向量 ID 到 skill_id 的映射)。
    4.  **分类索引**：**重要优化**。不要建一个大库，而是按 `category` 建立 **10-20 个小索引文件** (如 `index_climate.faiss`, `index_window.faiss`)。这将使检索速度提升 10 倍。

#### 3. 训练路由分类器 (Router Classifier)
训练一个超轻量的文本分类模型，用于判断用户指令属于哪个 `category`。
*   **模型**：TextCNN 或 DistilBERT (INT8 量化后 < 10MB)。
*   **训练数据**：从 2000 个技能的 `example_queries` 自动生成，标签为 `category`。
*   **输出**：概率分布 (e.g., `{climate: 0.9, window: 0.05, ...}`)。
*   **部署格式**：转换为 **ONNX** 或 **TFLite** 格式，以便在 8295 NPU/CPU 上极速运行。

### 第二阶段：端侧推理引擎开发 (C++/Rust 集成)

在车机端 (Android Automotive OS / Linux)，你需要开发一个高性能的推理服务。推荐使用 **C++** 结合 **llama.cpp** (用于 LLM) 和 **ONNX Runtime / QNN** (用于 Router/Embedding)。

#### 1. 环境搭建 (高通 8295)
*   **SDK**：安装 Qualcomm AI Engine Direct SDK (QNN)。
*   **后端**：
    *   **LLM**: 使用 `llama.cpp` (支持 Vulkan GPU 加速，或通过自定义后端接入 QNN)。
    *   **Router/Embedding**: 使用 `ONNX Runtime` 配置 `QNN Execution Provider`。
*   **量化**：
    *   LLM (Gemma 3n E2B): 量化为 **Q4_K_M** (4-bit)。
    *   Router/Embedding: 量化为 **INT8**。

#### 2. 核心流程代码逻辑 (伪代码)

```cpp
class CarControlEngine {
    RouterClassifier router;
    VectorSearcher searcher; // 包含多个 category 的 FAISS 索引
    LlmEngine llm;           // Gemma 3n E2B

    Response process(string user_text) {
        // Step 1: 规则匹配 (极速拦截)
        if (rule_matcher.match(user_text)) return rule_matcher.execute();

        // Step 2: 意图路由 (< 20ms)
        // 输入文本，输出最可能的 Category (如 "window_control")
        string top_category = router.predict(user_text); 
        
        // Step 3: 向量检索 RAG (< 50ms)
        // 只在 top_category 对应的子索引中搜索 Top 3 技能
        vector<Skill> candidates = searcher.search(user_text, top_category, top_k=3);
        
        if (candidates.empty() || candidates[0].score < 0.6) {
            return fallback_handler(user_text); // 兜底：云端或报错
        }

        // Step 4: 构造精简 Prompt
        string prompt = build_prompt(user_text, candidates);
        // Prompt 示例:
        // "User: '窗户开个缝'. Candidates: [1.id:win_slit, 2.id:win_half, 3.id:win_close]. Select ID and params."

        // Step 5: LLM 决策与参数提取 (< 300ms)
        // 启用 Grammar Constrained Decoding (强制输出 JSON)
        Json output = llm.generate(prompt, grammar=json_schema);

        // Step 6: 执行
        return execute_skill(output);
    }
};
```

#### 3. 关键优化技术
*   **Grammar Constrained Decoding (语法约束)**：
    *   在 `llama.cpp` 中使用 `gbnf` 文件限制输出必须是特定的 JSON 格式。
    *   **效果**：杜绝模型输出废话，直接得到 `{"skill_id": "...", "params": {...}}`。
*   **Context Caching (上下文缓存)**：
    *   将 System Prompt 和 车辆当前状态 (如“当前车速 60km/h”) 预先计算并缓存 KV Cache。
    *   每次请求只需计算用户输入部分，大幅降低 TTFT (首字延迟)。
*   **并行流水线**：
    *   ASR 流式输出时，每收到 5 个字就触发一次 Router 预测。当 ASR 结束时，Router 结果已就绪，直接进入检索。

### 第三阶段：测试与迭代 (闭环优化)

#### 1. 自动化测试集
构建包含 5000+ 条语料的测试集，覆盖：
*   **标准指令** (1000 条)：完全匹配技能描述。
*   **口语化指令** (2000 条)：带口音、省略、倒装 (如“冷死我了”、“窗开条缝呗”)。
*   **模糊/多义指令** (1000 条)：需要上下文判断 (如“打开这个”配合手势)。
*   **负样本** (1000 条)：非车控指令 (如“今天天气怎么样”)，测试拒答能力。

#### 2. 评估指标
*   **准确率 (Accuracy)**：Top 1 技能匹配正确率 > 95%。
*   **延迟 (Latency)**：P99 延迟 < 1000ms (目标 P95 < 800ms)。
*   **资源占用**：内存 < 3GB, NPU 利用率 < 80% (预留余量给其他应用)。

#### 3. Bad Case 分析与修复
*   **检索失败**：如果用户说法太偏，向量检索没召回。 -> **对策**：将该说法加入 `example_queries`，重新生成向量索引 (无需重训模型)。
*   **路由错误**：如果把“开窗”分到了“空调”类。 -> **对策**：将该样本加入 Router 训练集，微调分类器 (仅需几分钟)。
*   **LLM 幻觉**：选了错误的候选项。 -> **对策**：优化 Prompt，或在 Prompt 中加入更强的思维链 (CoT) 引导 (但在 2B 模型上需谨慎，可能增加延迟)。

---

### 第四阶段：OTA 动态更新机制

当车辆 OTA 升级新增 50 个技能时：
1.  **云端处理**：在云端生成新技能的向量，合并到对应的 `category_index.faiss`。
2.  **下发包**：仅下发增量索引文件和新的 `skills_database.json` 片段 (几 MB)。
3.  **车机热加载**：车机服务重启或动态加载新索引，**无需更新整个 APK 或重训模型**。
4.  **生效**：用户立即可用新技能。
